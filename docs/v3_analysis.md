**日期**: 2026-04-06  
**版本**: V3 → V4 重构

## 一、系统概述与需求梳理

### 1.1 系统目标

NoiseGuard 项目的核心目标是：**利用 SM-24 地震检波器，自动检测并记录楼上邻居产生的噪音事件（脚步、跺脚、拖拽家具、物体掉落等），并实时报警提醒用户**。

系统的核心价值主张是物理层面的信号分离——SM-24 地震检波器是速度/加速度传感器（非麦克风），灵敏度 28.8 V/(m/s)，自然频率 10Hz，有效范围 10-240Hz。安装在天花板时，它只响应通过楼板结构传导的机械振动，从物理上天然免疫室内空气传播的声音（说话、电视、音乐等）。

### 1.2 硬件配置

| 组件     | 规格                                              |
| -------- | ------------------------------------------------- |
| 传感器   | SM-24 地震检波器, 28.8V/(m/s), 375Ω, 10Hz自然频率 |
| 安装位置 | 天花板下表面（胶粘/螺丝固定）                     |
| 音频接口 | Dell P39F 笔记本 Realtek 板载声卡 3.5mm 麦克风口  |
| 采样率   | 44100Hz                                           |
| 帧长     | 1024 样本 (≈23ms)                                 |
| 电脑位置 | 天花板正下方约 2.5 米                             |

### 1.3 用户核心诉求

用户的需求可以用一句话概括：**楼上有动静（脚步、拖拽、掉东西）→ 准确报警；没动静 → 绝对不响**。

具体拆解：

1. **灵敏度**：真实的楼上活动（脚步声、物体掉落、家具拖拽）必须被准确检测到
2. **特异性**：自身活动（打字、滑触摸板、走动）、环境噪声（空调、冰箱、电梯）不得误报
3. **实时性**：检测到事件后立即报警（延迟 < 1秒），不要等太久才响
4. **可靠性**：7×24小时稳定运行，不漂移、不跑偏、不需要频繁重新校准

## 二、故障现象详述

用户报告了四个严重问题，严重性按用户感受排序：

### 2.1 假阴性（漏检）—— "有声音不响"

**现象**：楼上明显有人活动，包括：

- 物体掉落到地板（冲击型振动，理论上最容易检测）
- 明显的脚步声（周期性结构振动，检波器最擅长检测的信号）

系统**完全没有报警**。这是最严重的问题——系统存在的唯一理由就是检测这些事件，而它完全失职。

**初步分析**：如果 SM-24 正确安装在天花板并连接到正确的声卡输入，这些信号的能量应远超检波器底噪。漏检只有几种可能：

1. 输入设备选错（没有在读检波器的信号）
2. 校准数据异常导致阈值过高（不太可能，因为同时还有误报）
3. 报警静默机制误伤（误报触发后进入静默期，真实事件被屏蔽）

### 2.2 假阳性（误报）—— "没声音乱响"

**现象**：环境安静、楼上无人活动时，系统自行触发报警。

**初步分析**：与漏检同时出现的误报极其反常——说明系统不是"太灵敏"或"太迟钝"，而是**检测逻辑本身有根本性错误**，在错误的信号上触发、在真实的信号上沉默。

### 2.3 特定操作误触发 —— "滑触摸板报警"

**现象**：用户滑动笔记本触摸板时系统立即报警，但敲击键盘或拍打电脑外壳无反应。

**这是最关键的线索**。分析：

- 笔记本在天花板正下方 2.5 米处
- 手指滑动触摸板产生的机械能极微弱，不可能通过 2.5 米的空气+结构传导到天花板的检波器
- 但触摸板的**电容扫描电路**会产生 50-200Hz 的周期性电磁干扰（EMI）
- 如果检波器电缆未屏蔽或靠近笔记本，EMI 会直接耦合到信号线
- 这种干扰是**类正弦的平滑信号**，频率恰好落在检波器的检测频段内（5-250Hz）

替代可能性：如果输入设备实际是笔记本内置麦克风（而非 SM-24），触摸板摩擦音会被近距离拾取。

### 2.4 模式矛盾 —— 为什么键盘不触发？

这进一步证实了 EMI 假说：

- **触摸板**：电容扫描 → 50-200Hz 周期性 EMI → 频率在检测频段内 → 被识别为"振动"
- **键盘**：USB HID 通信 → MHz 级别 EMI → 带通滤波器(5-250Hz)直接滤除
- **拍打电脑**：空气声 → 检波器不响应结构传导

## 三、根因分析（三重根因）

### 3.1 第一根因：输入设备配置可能错误

**当前 config.json 设备配置**：

```json
{
  "device": {
    "index": 1,
    "name": "本机麦克风 (适用于数字麦克风的英特尔® 智音技术)",
    "sample_rate": 44100,
    "block_size": 1024
  }
}
```

**这是 Intel Smart Sound 数字麦克风——笔记本的内置麦克风，不是 SM-24 检波器！**

如果系统一直在用笔记本内置麦克风而非 SM-24：
| 现象 | 用内置麦克风的解释 | 用SM-24的解释 |
|------|-------------------|--------------|
| 触摸板报警 | ✅ 近距离摩擦音被拾取 | ⚠ EMI耦合(需验证) |
| 键盘不报警 | ✅ 按键音低于阈值 | ✅ 不产生楼板振动 |
| 楼上脚步漏检 | ✅ 麦克风拾取结构振动能力差 | ❌ 不应该漏检 |
| 掉东西漏检 | ✅ 空气传导衰减大 | ❌ 不应该漏检 |
| 安静时误报 | ✅ 噪底极低,微小扰动触发 | ⚠ EMI/量化噪声 |

**内置麦克风假说能完美解释所有现象**。SM-24 假说需要额外的 EMI 条件，且无法解释脚步/掉东西漏检。

**结论**：**输入设备极可能配置错误**，系统在读笔记本麦克风而非检波器。用户需要在设备列表中选择 SM-24 所连接的模拟输入端口（通常显示为 "Microphone (Realtek Audio)" 或 "Line In"）。

### 3.2 第二根因：校准数据严重异常

**当前校准数据**：

```json
{
  "calibration": {
    "calibrated": true,
    "noise_floor_db": -85.04,
    "noise_band_power": [1e-10, 1e-10, 1e-10]
  }
}
```

三个致命问题：

**问题 A：noise_floor_db = -85.04dB 极度异常**

正常场景的参考值：

- SM-24 + 板载声卡：底噪约 -60 ~ -50 dBFS
- 笔记本内置麦克风：底噪约 -55 ~ -45 dBFS
- 全数字静音（无输入设备）：约 -90 ~ -80 dBFS

-85dB 接近数字静音水平，说明校准时设备几乎没有输入信号。可能原因：

1. 校准时使用了错误的设备（没有实际输入）
2. SM-24 未正确连接（线松了/断了）
3. 声卡输入增益为零

**问题 B：noise_band_power 全部是 1e-10（最小钳位值）**

代码中 `_min_noise_power = 1e-10` 是防止除零的安全下限，对应约 -100dBFS。三个频段全部触底意味着：**校准期间每个频段的信号功率都低于 -100dB，即完全无信号**。

这直接导致了灾难性后果：频谱减除变成了 `signal - 0 = signal`，任何微小信号都会产生巨大的 excess_db 值。

**问题 C：sensitivity_db = 1**

用户或系统设置的灵敏度阈值仅 1dB——意味着信号只需超过噪声基线 1dB 就触发报警。结合 -85dB 的底噪，实际触发门限为 -84dBFS。这个门限低得**荒谬**：

- 热噪声/量化噪声的随机波动就能超过 1dB
- 任何 ADC 的固有噪声在特定频段都可能超过 -84dBFS
- 触摸板 EMI 或电源纹波轻松超过

**这就是"安静时乱响"的直接原因**：系统在 -84dBFS 的门限上运行，几乎任何电气噪声都能触发。

### 3.3 第三根因：检测算法自身存在多个设计缺陷

即使设备正确、校准正常，当前算法仍有根本性问题。详见第四章。

## 四、当前算法逐层剖析与缺陷

### 4.1 信号处理链审计

当前处理链：

```
raw PCM → 4阶Butterworth带通(5-250Hz) → 50Hz谐波陷波(×4) → 分析
```

**评价**：信号处理链本身设计合理，问题不在这里。带通滤波有效隔离了 SM-24 的工作频段，陷波器消除了 50Hz 电源干扰。

### 4.2 功率检测双路径分析

当前使用双路径检测：

1. **持续路径**：EMA 平滑功率 > 2× 噪声基线 → excess_sustained
2. **瞬态路径**：原始帧功率 > 4× 噪声基线 → excess_transient

**缺陷 A：阈值倍数是相对的，底线太低时形同虚设**

2× 和 4× 是相对于 noise_power 的倍数。当 noise_power = 1e-10（校准错误导致）时：

- 持续阈值 = 2e-10，对应 -97dBFS
- 瞬态阈值 = 4e-10，对应 -94dBFS

任何信号都能轻松超过这个门限。

**缺陷 B：没有绝对能量下限**

系统完全依赖相对阈值（信噪比），没有类似"信号低于 -70dBFS 绝对不触发"的硬限制。这意味着一旦校准值异常偏低，系统就会对极微弱的噪声触发。

### 4.3 自适应基线的单向漂移问题

当前自适应基线使用非对称 EMA：

```python
if overall_power < self._adapt_overall_power:
    alpha = 0.01    # 下行较快
else:
    alpha = 0.001   # 上行极慢（慢100×）
```

**缺陷：基线只降不升，长时间运行后必然偏低**

这个设计意图是"防止真实噪声被吸收进底噪"——如果基线跟着噪声一起升高，就会永远检测不到持续噪声。但实际效果是灾难性的：

1. **启动时如果环境稍嘈杂**（HVAC/电梯/短暂施工），基线被设到较高值
2. **环境安静后**，基线以 α=0.01 缓慢下降
3. **经过几小时**，基线会降到远低于当前噪声水平的值
4. **正常的环境背景噪声**（如建筑结构热膨胀、远处交通微振动）开始超过基线 → 误报

更严重的是：基线下降速度 (α=0.01) 是上升速度 (α=0.001) 的 10 倍。这意味着基线有**系统性向下偏移**。运行时间越长，偏移越严重，误报越多。

**正确做法**：应该使用百分位追踪（取最近 N 秒功率的第 20 百分位作为基线），自动排除事件帧，同时允许控制性双向更新。

### 4.4 STA/LTA 在极低信噪比下的失效

STA/LTA（短时/长时平均比）是地震学经典算法，用于在稳定背景中检测突发事件。但当应用于 excess 能量（减除噪声后的超额功率）时：

- 安静时，excess ≈ 0 + 极小随机波动
- STA 和 LTA 都极小（~1e-20 量级）
- 一个微小的随机脉冲（如 ADC 量化噪声跳变）可以让 STA/LTA = 100+
- 算法误判为"地震"式的突发事件

**本质问题**：STA/LTA 假设背景信号有稳定的非零能量。当 excess ≈ 0 时，STA/LTA 的比值变得统计不可靠（分母趋近于零）。

### 4.5 分类器过于宽松

当前 `classify_vibration()` 的入口判断：

```python
if a['excess_db'] < sensitivity_db:
    return 'silent', '安静'
```

sensitivity_db = 1 时，excess_db ≥ 1 的任何信号都通过。之后的分类逻辑全部基于频谱特征（sub_ratio、mid_ratio、crest 等），但**完全没有考虑信号的统计特征**。

具体问题：

- **没有峰度（kurtosis）判断**：真实的物理冲击有极高的峰度（信号中有尖锐脉冲），而 EMI/触摸板噪声是平滑的类正弦信号（低峰度）。当前算法无法区分这两种截然不同的信号。
- **is_impact 判断依赖帧间跳变**：`jump_db > 4 and sta_lta > 2.0`。但帧间跳变也可能来自 EMI 的开关（触摸板开始/停止滑动），而不是真实冲击。
- **分类兜底过于宽松**：`crest > 2.0` 就判为 `impact`。大量非冲击信号也有 crest > 2.0（如脉冲型 EMI）。

### 4.6 状态机的触发条件问题

```python
if is_impact:
    # 冲击只有1-2帧, 必须立即触发
    self._enter_active(a, cls, now)
```

对 is_impact 直接触发、不走确认计数，目的是快速响应真实冲击。但当 classify_vibration 错误地将 EMI 脉冲标记为 `impact` 时，这条路径导致**一帧误报即触发报警**——连确认窗口的保护都没有。

结合 `_alert_mute_sec` 静默机制：误报触发 → 进入静默 N 秒 → 静默期内真实事件被忽略 → "有声音不响"。

**这完美解释了漏检和误报同时存在的矛盾现象**：EMI/噪声触发误报 → 进入静默 → 真实事件恰好在静默期内发生 → 被屏蔽 → 用户看到"乱响+不响"的组合。

## 五、V3 重设计方案

### 5.1 设计哲学：峰度驱动的多层门控

V3 的核心设计哲学基于一个关键物理洞察：

> **真实的楼上振动（脚步、冲击、拖拽）在信号统计特征上与电气干扰（EMI、触摸板噪声、量化噪声）有本质区别。**

具体而言，信号的**峰度（Kurtosis）**是最佳判别指标：

| 信号类型       | 峰度值   | 波形特征           |
| -------------- | -------- | ------------------ |
| 高斯白噪声     | ≈3.0     | 随机波动，无尖峰   |
| 正弦波/EMI     | ≈1.5-2.5 | 平滑周期，极少尖峰 |
| 触摸板电容扫描 | ≈2.0-3.5 | 近似正弦的周期干扰 |
| 建筑微振动     | ≈3.0-4.0 | 低能随机           |
| 走路/拖拽      | ≈4.0-8.0 | 脉冲序列叠加连续   |
| 跺脚/掉东西    | ≈8.0-50+ | 极尖锐脉冲         |

**V3 的核心策略：信号的峰度值动态调节检测阈值。**

- 高峰度信号（真实冲击）→ 使用标准灵敏度阈值
- 中等峰度信号（走路/拖拽）→ 使用 1.5× 灵敏度阈值
- 低峰度信号（EMI/环境噪声）→ 使用 3× 灵敏度阈值

这一策略的优雅之处在于：**对真实楼上事件的检测性能不受任何影响**（它们天然高峰度），而 EMI 和环境噪声被自动抑制（仅在极高能量时才可能穿过 3× 阈值）。

### 5.2 三级门控架构

V3 采用三级门控，每一级独立验证，全部通过才触发：

**第一级：能量门控（现有，优化）**

- excess_db = 超出噪声基线的 dB 数
- 必须 > 0 才进入下一级
- 噪声基线使用改进的百分位自适应追踪

**第二级：峰度门控（新增，核心创新）**

- 计算每帧信号的峰度值
- 根据峰度动态调整有效阈值：
  - `kurt >= 6.0` → `threshold = sensitivity_db`（标准）
  - `4.0 <= kurt < 6.0` → `threshold = sensitivity_db × 1.5`
  - `kurt < 4.0` → `threshold = sensitivity_db × 3.0`
- excess_db 必须超过动态阈值才进入下一级

**第三级：时序门控（状态机，现有，保留）**

- 持续事件：确认帧数（confirm_frames）
- 冲击事件：**不再无条件单帧触发**，要求 `kurt > 8` 才允许快速触发
  - 这防止了 EMI 脉冲通过冲击路径误触发
- 进入 active 后的冷却期、录音、报警逻辑保持不变

### 5.3 改进的百分位自适应基线

**V2 问题**：非对称 EMA（只降不升）导致基线系统性偏低。

**V3 方案**：百分位追踪 + 受控双向更新

```
原理:
1. 维护最近 30 秒的功率值环形缓冲区
2. 安静时, 取第 20 百分位作为基线参考
3. 基线向参考值双向移动, 但速度不同:
   - 下降: α=0.02 (约 2.3 秒时间常数)
   - 上升: α=0.005 (约 9.2 秒时间常数)
4. 事件期间 (STA/LTA > 1.3 或 excess > 1dB) 暂停更新
```

**为什么用百分位而非 EMA**：

- EMA 对离群值敏感：一次强冲击可以把 EMA 拉高，很久才恢复
- 百分位天然鲁棒：第 20 百分位自动忽略最高的 80% 值（即事件帧），只反映安静时的真实底噪
- 30 秒窗口足够长以覆盖正常的环境波动周期，又足够短以跟踪缓慢变化（如温度导致的 ADC 漂移）

**为什么允许上升**：

- V2 的"只降不升"在某些场景下是必要的（防止持续噪声被吸收），但长期运行会失败
- V3 的慢上升（α=0.005，比下降慢 4×）是折中方案：
  - 正常的环境变化（空调启停、交通潮汐）：基线在分钟尺度上跟踪，不会导致漏检
  - 真正的持续楼上噪声：因为事件期间暂停更新，噪声不会被吸收

### 5.4 瞬态/冲击检测升级

**V2 方案**：帧间 excess_db 跳变 > 4dB 且 STA/LTA > 2.0
**V3 方案**：保留帧间跳变检测，但增加峰度约束

```
V2: is_impact = (jump_db > 4) AND (sta_lta > 2.0)
V3: is_impact = (jump_db > 4) AND (sta_lta > 2.0) AND (kurtosis > 5)
```

添加 `kurtosis > 5` 的理由：

- 真实的物理冲击（掉东西/跺脚）在 23ms 帧内有极尖锐的脉冲 → 峰度 > 10
- EMI 开关瞬变也能产生帧间跳变 > 4dB + STA/LTA > 2.0
- 但 EMI 瞬变是平滑的上升/下降沿 → 峰度 < 4
- 峰度 > 5 的约束有效排除 EMI 而不影响真实冲击

### 5.5 分类器 V3 —— 峰度感知分类

```
classify_vibration_v3(analysis, sensitivity_db):

    # 第一步: 计算峰度自适应有效阈值
    if kurtosis >= 6.0:
        effective_threshold = sensitivity_db       # 高峰度: 真实冲击
    elif kurtosis >= 4.0:
        effective_threshold = sensitivity_db × 1.5 # 中峰度: 走路/拖拽
    else:
        effective_threshold = sensitivity_db × 3.0 # 低峰度: 可能是干扰

    # 第二步: 能量门控
    if excess_db < effective_threshold:
        return '安静'

    # 第三步: 分类 (仅对通过门控的信号)
    高峰度 + STA/LTA飙升 → '冲击'
    周期性振动 → '脚步'
    中频为主 + 低峰度 → '拖拽'
    超低频主导 + 高能量 → '持续振动'
    其他 → '轻微振动'
```

**关键改进**：

- `excess_db < sensitivity_db × 3` 的低峰度信号直接过滤（EMI 免疫）
- 低峰度 + 低能量组合不再有机会触发
- 冲击分类需要峰度支持（`kurt > 6`），不再仅靠帧间跳变

### 5.6 校准验证机制

V3 在校准完成后自动验证结果合理性：

1. **底噪过低警告**：`noise_floor_db < -80` → 提示用户检查设备连接
2. **频段功率全部触底**：所有 `noise_band_power ≤ 1e-9` → 提示设备可能无信号
3. **底噪过高警告**：`noise_floor_db > -30` → 提示可能有持续性干扰或增益过高

校准失败时不保存结果，保留之前的有效校准数据。

### 5.7 报警静默期改进

V2 的静默机制是问题链的关键环节（误报 → 静默 → 漏检）。V3 改进：

- **快速触发不再无条件**：冲击路径需要 `kurt > 8`，EMI 脉冲走不了快速通道
- **静默期内强事件穿透**（已有，保留）：真正的重击（excess > 2× threshold）可以打破静默
- **减少误报 → 自然减少不必要的静默 → 漏检概率下降**

## 六、整体参数调优建议

### 6.1 灵敏度 (sensitivity_db)

| 值    | 适用场景                          |
| ----- | --------------------------------- |
| 1-3   | ❌ 极度不推荐，几乎必然误报       |
| 4-6   | 高灵敏度，适合安静环境 + 正确校准 |
| 6-10  | **推荐默认值**，平衡灵敏/误报     |
| 10-15 | 中等灵敏，适合噪声环境            |
| 15-20 | 低灵敏，只检测强冲击              |

**V3 默认值**：6（从配置中的 1 改为 6，提供合理的初始体验）

### 6.2 确认帧数 (confirm_frames)

当前值 2 帧（≈46ms）偏少。对于非冲击事件，建议 3-4 帧（≈70-90ms），能有效过滤单帧随机波动。

冲击事件在 V3 中通过峰度门控保护，即使走确认流程也有快速通道（`kurt > 8` 仅需 1 帧确认）。

### 6.3 冷却时间 (cooldown_seconds)

当前 2 秒合理。配合改进的静默机制，减少误报后自然减少不必要的冷却。

## 九、总结

NoiseGuard 的检测失败是**三重根因叠加**的结果：

1. **输入设备可能错误**（用笔记本麦克风代替检波器） — 这是最可能的主因
2. **校准数据异常**（底噪-85dB + 频段功率全触底） — 导致阈值形同虚设
3. **算法缺乏鲁棒性**（无峰度判别 + 单向漂移基线 + 过低默认灵敏度） — 即使硬件正确也会有问题

V3 通过**峰度驱动的多层门控**从根本上解决第三点，同时通过**校准验证**和**合理默认值**缓解第二点的影响。第一点需要用户在部署时确认设备选择。

三个改进独立生效、相互补强：

- 峰度门控 → 区分真实振动 vs EMI/噪声 → **核心改进**
- 百分位基线 → 防止长期漂移 → **稳定性改进**
- 校准验证 → 捕获配置错误 → **可靠性改进**

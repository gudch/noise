# NoiseGuard V4 — 需求分析与重构设计报告

**日期**: 2026-04-06  
**背景**: V3 实测仍存在"大声不响、安静误报"问题，需要从用户实际需求出发彻底重构检测逻辑

---

## 一、用户需求完整梳理

### 1.1 用户的核心诉求（一句话）

**楼上发出了我能感知到的噪音 → 立刻报警；没噪音 → 绝对不响。**

### 1.2 用户明确表达的需求

| #   | 原话摘要                                               | 需求提炼                                                 |
| --- | ------------------------------------------------------ | -------------------------------------------------------- |
| 1   | "有声音不响没声音乱响"                                 | 系统的第一优先级是**不漏报**，第二优先级是**不误报**     |
| 2   | "很大的声音比如放东西、放桌椅板凳、大力走步反而不告警" | 高能量事件必须100%触发，这是最基本的底线                 |
| 3   | "很轻的脚步我们都听不到的能识别"                       | 灵敏度调节应该人性化，不该检测到人都听不到的微弱振动     |
| 4   | "何必区分哪些是脚步声？何必加这个逻辑？"               | **不需要分类**。用户不关心楼上在干什么，只关心有没有噪音 |
| 5   | "duang一声把人惊醒的噪音"                              | 强冲击必须立即报警，零延迟                               |
| 6   | "走路声音很大且持续很长"                               | 持续性噪音也是噪音，必须报警                             |
| 7   | "钢珠掉地上、小孩跑跳、拍皮球、切菜剁菜、放水桶"       | 任何类型的振动噪音都应报警，不遗漏                       |
| 8   | "任何一种能让我听到的都是噪音"                         | 检测阈值应与人的感知阈值对齐                             |
| 9   | "步态周期、冷却时间很影响程序判断逻辑"                 | 删除步态周期性检测和冷却时间机制                         |
| 10  | "确认帧数功能我都不知道啥意思"                         | 删除或大幅简化确认帧逻辑                                 |

### 1.3 从用户心理角度分析

用户的使用场景是**居家生活场景**，他们的心理模型非常简单：

1. **他能听到楼上的声音** → 系统应该同时报警
2. **他听不到任何声音** → 系统不应该响
3. 如果系统比他"耳朵还灵"（检测到他听不到的声音）→ 觉得**过于灵敏/误报**
4. 如果系统在他明显听到噪音时不响 → 觉得**这程序是废物**

用户**不关心**：

- 楼上在做什么（走路/跑步/拖拽/跺脚）→ 不需要分类
- 技术指标（峰度/STA-LTA/频谱）→ 不需要在界面上显示
- 学术级精准度 → 只需要和人耳感知一致

### 1.4 从实际场景角度分析

**场景1：白天在家工作**

- 楼上偶尔走路、搬东西 → 报警提醒（用于记录取证）
- 自己打字、喝水、走动 → 不报警（SM-24物理隔离，这个天然能做到）

**场景2：夜间睡觉**

- 楼上突然duang一声 → 必须立刻报警
- 安静的夜晚 → 绝对不能响（否则影响睡眠，系统比楼上邻居更烦）

**场景3：长时间噪音**

- 楼上持续走来走去半小时 → 应该报警一次（记录事件），不需要每两步响一次
- 楼上装修持续敲墙 → 报警一次就够了

### 1.5 为用户考虑的额外注意事项

| 考虑点             | 说明                                        | 设计决策                                                 |
| ------------------ | ------------------------------------------- | -------------------------------------------------------- |
| 报警疲劳           | 如果同一次持续活动每隔2秒响一次，用户会抓狂 | 保留合理的报警间隔（但不是"冷却时间"那种阻断检测的机制） |
| 报警静默期的副作用 | V3的静默期会屏蔽真实事件                    | V4改为：报警后不停止检测，只是不重复播放报警音           |
| 确认帧的副作用     | 确认帧导致短促冲击被延迟或遗漏              | V4删除确认帧，改为信号超阈值即触发                       |
| 灵敏度与人耳对齐   | sensitivity_db=1太低                        | 默认sensitivity_db=6，让用户可调但给个合理默认值         |
| 长期运行稳定性     | 基线漂移导致几小时后开始误报                | 保持百分位基线追踪（V3已有），这个是好的                 |

## 二、V3 现存问题诊断

### 2.1 用户反馈："大声不响、小声反而响"

**根因分析**：

V3 的检测逻辑有过多的"门"在拦截信号：

```
信号 → 功率检测 → 峰度门控 → classify_vibration分类 → 状态机(confirm_frames) → 冷却时间 → 报警静默期
```

每一道门都可能把真实事件挡住：

1. **classify_vibration 的峰度门控**：`kurt < 4 → eff_sens = sensitivity_db × 3`。如果一个大声的持续活动（搬桌子、拖椅子）的峰度在3-4之间（因为是持续摩擦型，不是脉冲型），阈值被放大3倍 → 不触发
2. **confirm_frames = 2**：需要连续2帧都被分类为非silent才触发。但一个短促的冲击（duang一声）可能只有1帧超阈值 → 被confirm_frames拦截
3. **冷却时间 cooldown_seconds = 2**：事件结束后2秒内又来新事件，状态机认为是同一事件的延续，不重新触发
4. **报警静默期 min_interval_seconds = 3**：报警后3秒内（用户设的值）不报警 → 连续事件的第二次被屏蔽
5. **\_tick()中的kurt>8要求**：只有 kurtosis > 8 的冲击才能1帧触发，否则走confirm_frames。真实的搬重物撞地面 kurt 可能只有5-7 → 被要求走2帧确认 → 单次冲击就1帧 → 漏检

### 2.2 为什么轻微脚步反而能触发？

轻微但持续的脚步声 → 多帧连续超阈值 → 轻松通过 confirm_frames = 2 → 触发。而重物砸地一次 → 只有1帧超阈值 → confirm_frames 要求2帧 → 不触发。

**这是 confirm_frames 机制的本质矛盾**：它保护了误报，但同时杀死了瞬态事件的检测能力。

### 2.3 步态周期性检测的问题

`_detect_periodicity()` 用自相关法检测0.3-1.5秒的周期性。这个功能：

- 增加了计算开销
- 结果只用于分类（在classify_vibration中）
- 用户明确说不需要分类
- 没有任何正面作用，只有副作用（代码复杂度）

### 2.4 分类器的问题

`classify_vibration()` 返回 impact/step/drag/rumble 等类别。用户明确表示：

> "何必区分哪些是脚步声？区分出来干啥呢？"

分类器做的事情：

1. 按峰度调整阈值 → 这个有用但方式错误（不应该恶化对真实信号的灵敏度）
2. 区分事件类型 → 用户不需要
3. 增加了多个分支条件 → 每个分支都可能漏检或误判

## 三、V4 重构设计

### 3.1 设计哲学

**极简主义 + 与人耳对齐**：

1. **只做一个判断**：当前噪音是否超过阈值？超过 → `noise`（噪音），不超过 → `silent`（安静）
2. **不做分类**：不区分脚步/冲击/拖拽，统一叫"噪音"
3. **不做步态分析**：删除周期性检测
4. **不做确认帧**：信号超阈值即触发，不等第二帧
5. **智能报警间隔**：报警后一段时间内不重复报警音，但**不停止检测**（和V3的"静默期暂停检测"本质不同）
6. **峰度只用于抗干扰**：不用峰度区分事件类型，只用于判断"这个信号是不是电气干扰"

### 3.2 新检测流程（极简）

```
信号 → 带通滤波 → 陷波 → 计算功率和峰度
                          ↓
                    功率 > 噪声基线 × 阈值？
                          ↓
                    是 → 峰度 > 3.5？（排除纯电气干扰）
                          ↓
                    是 → 状态 = "noise" → 触发报警
                          ↓
                    否 → 状态 = "silent"
```

对比 V3 的流程：

```
V3: 信号 → 滤波 → 功率 → 峰度分级阈值 → 分类(5种) → 状态机(3态+confirm) → 冷却 → 静默 → 报警
V4: 信号 → 滤波 → 功率 → 超阈值？ → 非干扰？ → 报警
```

### 3.3 状态机简化

**V3 状态机**：`silent → (confirm_frames) → active → cooldown → silent`  
**V4 状态机**：`silent ↔ noise`

```
silent状态:
  收到超阈值帧 → 切换到noise → 触发报警

noise状态:
  收到超阈值帧 → 保持noise（更新峰值等）
  收到低于阈值的帧 → 累计quiet_count
  quiet_count > quiet_frames(约0.5秒) → 切换到silent → 记录事件结束
```

**关键改变**：

- 删除 confirm_frames：超阈值即触发，1帧就够
- 删除 cooldown 状态：噪音停了就是停了，signal说了算
- 报警间隔通过前端控制（不重复播放音效），后端不阻断检测

### 3.4 报警机制改造

V3的问题：报警 → 进入静默期(N秒) → 静默期内不检测 → 漏报

V4的改造：

- 后端**始终检测**，不暂停
- 报警音播放间隔由前端控制：上次播放后N秒内不重复播放
- 事件仍然正常记录到数据库（即使不重复播放报警音）
- 前端卡片/状态始终实时显示当前状态

### 3.5 classify_vibration 简化

```python
# V4: 只做二分类
def classify_vibration(a, sensitivity_db):
    excess = a['excess_db']
    kurt = a.get('kurtosis', 3.0)

    # 阈值: 就是灵敏度
    if excess < sensitivity_db:
        return 'silent', '安静'

    # 峰度过滤: kurtosis < 3.5 的信号大概率是电气干扰(EMI/触摸板)
    # 真实物理振动在能量超阈值时 kurtosis 至少 > 3.5
    if kurt < 3.5 and excess < sensitivity_db * 2:
        return 'silent', '安静'

    return 'noise', '检测到噪音'
```

**为什么保留峰度3.5的检查**：

- 这不是"分类"，而是纯粹的抗干扰
- 触摸板EMI/正弦干扰的峰度 ≈ 1.5-3.0，所以 <3.5 的低能量信号大概率是干扰
- 但如果能量超过 sensitivity_db × 2（很强的信号），即使峰度低也让它通过（安全阀）
- 真实楼上振动在超阈值后几乎不可能 <3.5

### 3.6 删除的功能

| 删除项                  | 原理由               | 删除原因                                 |
| ----------------------- | -------------------- | ---------------------------------------- |
| `_detect_periodicity()` | 检测步态0.3-1.5s周期 | 用户不需要分类，增加复杂度               |
| `_env_buf`              | 步态检测的包络缓冲   | 随步态检测一起删除                       |
| confirm_frames          | 要求连续N帧才触发    | 导致单次冲击漏检                         |
| cooldown状态            | 噪音停止后等N秒      | 无意义的延迟，可能导致新事件与旧事件合并 |
| 报警静默期(暂停检测)    | 防止重复报警         | 改为前端控制播放间隔，后端不阻断         |
| classify输出5种类别     | 区分脚步/冲击/拖拽等 | 用户不需要，统一为noise                  |
| periodicity字段         | 传给前端显示         | 随步态检测一起删除                       |
| jump_db / is_impact     | 帧间跳变检测         | 不再需要区分冲击类型                     |

### 3.7 保留的功能

| 保留项               | 原因                    |
| -------------------- | ----------------------- |
| 带通滤波(5-250Hz)    | SM-24物理特性决定，必须 |
| 50Hz陷波滤波         | EMI消除，必须           |
| 峰度计算             | 抗干扰用（不是分类用）  |
| STA/LTA              | 有参考价值，前端可显示  |
| 百分位自适应基线(V3) | 长期运行稳定性保障      |
| 录音功能             | 取证用                  |
| 会议麦辅助           | 可选功能，不影响主逻辑  |
| FFT频谱显示          | 前端图表展示用          |

### 3.8 默认参数调整

| 参数             | V3值     | V4值     | 原因                                           |
| ---------------- | -------- | -------- | ---------------------------------------------- |
| sensitivity_db   | 1        | 6        | 1dB太低，量化噪声都能触发；6dB约为人耳感知阈值 |
| confirm_frames   | 2        | 删除     | 导致瞬态漏检                                   |
| cooldown_seconds | 2.0      | 删除     | 改用quiet_frames(约0.5秒静默即结束事件)        |
| alert_mute机制   | 暂停检测 | 前端去重 | 后端始终检测不阻断                             |

## 六、总结

V4 的核心理念：**不要比用户更聪明**。

用户不需要系统告诉他楼上在走路还是在跑步还是在搬东西——他只需要知道"楼上有没有发出噪音"。系统的工作就是当振动超过阈值时报警，仅此而已。

所有额外的"智能"（分类、步态分析、峰度分级阈值、确认帧、冷却时间）都在做同一件事：**提高漏报率**。每多一道判断就多一个可能挡住真实事件的门。

V4 把这些门全部拆掉，只留一道：**信号功率 > 阈值 + 不是电气干扰 → 报警**。简单、可靠、符合用户预期。
